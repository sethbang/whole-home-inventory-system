# Security Policy

## Reporting a Vulnerability

The WHIS team takes security vulnerabilities seriously. We appreciate your efforts to responsibly disclose your findings.

### How to Report a Vulnerability

1. **DO NOT** create a public GitHub issue for security vulnerabilities
2. Email security concerns to [security@example.com]
3. Include detailed information about the vulnerability
4. Provide steps to reproduce if possible
5. Include any potential impact assessment

We will acknowledge receipt of your vulnerability report within 48 hours and send a more detailed response within 96 hours indicating the next steps in handling your report.

## Security Model

WHIS is a self-hosted single-tenant household inventory system. Its threat model assumes the deployment is on a trusted LAN or fronted by a reverse proxy/VPN. Anything beyond that requires additional hardening the application itself does not provide (see _Known security limitations_ below).

### Core Security Principles

1. **Data Privacy**
   - All data stored locally on the operator's host
   - No cloud synchronization
   - Backups are plain zip archives (not encrypted at rest — encrypt the host-side volume or the backup destination if that matters for your threat model)

2. **Network Security**
   - HTTPS-only communication in both dev and prod (self-signed CA during development; operator-provided TLS at the reverse-proxy layer in production)
   - CORS defaults restrict origins to `https://` schemes only
   - Local network by default

3. **Authentication**
   - JWT-based authentication via **PyJWT** (HS256 by default — change `ALGORITHM` in settings if you need RS/ES)
   - Password hashing with **bcrypt** via passlib (`bcrypt>=3.2,<4` pinned for passlib 1.7.4 compatibility)
   - Token expiry controlled by `ACCESS_TOKEN_EXPIRE_MINUTES` (default 30)
   - `BYPASS_AUTH` env flag for local development — refuses to load without an explicit `SECRET_KEY` otherwise

4. **Data Protection**
   - SQLite with Alembic-managed schema
   - Image uploads validated by Pillow magic-byte inspection (not trusted `Content-Type`)
   - Upload size cap via `MAX_UPLOAD_BYTES` (default 10 MB) and dimension cap via `MAX_IMAGE_DIMENSION` (default 8000 px)
   - Global exception handler redacts stack traces from HTTP responses unless `DEBUG=true` (dev only)

## Security Features

### Authentication System

- **Password Requirements (current):** WHIS does not currently enforce complexity, minimum length, common-password checks, or login rate limiting. This is deferred work — operators should put the service behind a reverse proxy with rate limiting (Synology DSM, Caddy, Traefik, etc.) if exposed beyond a trusted LAN.
- **Bypass mode:** `BYPASS_AUTH=true` short-circuits JWT verification and returns the hardcoded `DEV_USER`. The settings module refuses to load with `BYPASS_AUTH=false` and a missing/placeholder `SECRET_KEY`, so there's no default that leaks real auth off.

### Data Storage

- **Database:** SQLite at `settings.DATABASE_URL` (defaults to `backend/database/whis.db`). Alembic is the sole source of truth for schema (the pre-2.0.0 `Base.metadata.create_all()` path was removed).
- **File Storage:** Uploads go to `settings.UPLOAD_DIR`. Filenames are server-generated (`{timestamp}_{uuid4}{ext}`). Format whitelist: JPEG / PNG / WebP / HEIC. No malware scanning is performed.

### Network Security

- **HTTPS:** Enforced in both dev and prod. The dev certs are generated by `frontend/scripts/generate-certs.js` (see `certs/CERTIFICATE-SETUP.md`). For prod HTTPS, terminate TLS at your reverse proxy — the nginx container in `docker-compose.nas.yml` listens on HTTP 80 and expects upstream TLS termination.
- **CORS:** Configured via `CORS_ORIGINS` (HTTPS-only by default). Set a permissive list during LAN-only use; tighten when exposing externally.
- **Rate Limiting / WAF / HSTS:** Not provided by the application. Terminate and enforce at the reverse proxy layer.

## Known Security Limitations

1. **Local Network**
   - Designed for local network use
   - Remote access requires VPN or reverse proxy with operator-provided TLS + rate limiting
   - No built-in abuse protection

2. **Authentication**
   - Single user-role model (no role-based access control)
   - No OAuth / OIDC / MFA integration
   - No login attempt rate limiting
   - No password complexity enforcement
   - No refresh-token flow — sessions hard-expire after `ACCESS_TOKEN_EXPIRE_MINUTES`

3. **Backup Security**
   - Backups are plain zip archives (not encrypted at rest)
   - Manual restore process
   - Local storage only

## Security FAQs

**Q: Is my data stored in the cloud?**
A: No, WHIS stores all data locally on your host.

**Q: Can I access WHIS remotely?**
A: Yes, but only via VPN (e.g., Tailscale) or a reverse proxy (Synology DSM, Caddy, Traefik). The application does not include rate limiting, WAF, or abuse protection of its own.

**Q: How are passwords stored?**
A: Passwords are hashed with **bcrypt** via passlib. Configure rounds by swapping the passlib `CryptContext` arguments in `backend/app/security.py`.

**Q: Are images stored securely?**
A: Image uploads are validated by Pillow magic-byte inspection (not trusted `Content-Type`), enforced against a size cap and dimension cap, and stored with server-generated filenames. There is no malware scanning — if you accept uploads from untrusted parties, front the service with a scanning proxy.

**Q: What happens if `SECRET_KEY` leaks?**
A: All issued JWTs become forgeable. Rotate immediately (`python -c "import secrets; print(secrets.token_urlsafe(64))"`), update `.env`, and restart the backend. All existing sessions will be invalidated since token signatures no longer verify.
